# Agent Tool Bench

一个面向普通用户、开发者和 AI 工具爱好者的 AI Agent 工具评测项目。

如果你是从视频、文章或朋友推荐来到这里，这个仓库想帮你回答一个很朴素的问题：

> 面对越来越多的 AI Agent 工具，我到底该怎么判断它们好不好用、适不适合我？

这里不会只做“哪个工具更强”的主观排名，而是尝试用公开任务、可复现过程和清晰解释，观察不同工具在真实使用场景里的表现。

## 你可以在这里看到什么

- **评测任务**：我们会把任务背景、输入材料、成功标准和限制条件写清楚。
- **评测结果**：尽量区分事实表现、主观体验和使用建议。
- **复现说明**：你可以按同样的任务和条件重新跑一遍，验证结论是否成立。
- **工具档案**：记录不同 AI Agent 工具适合什么场景、不适合什么场景。
- **方法论**：解释为什么这么评测，以及这些维度对普通用户有什么意义。

## 我们关心的维度

评测不会只看“答案对不对”。一个 Agent 工具是否值得使用，还要看它在完整工作流里的表现。

| 维度 | 关注点 |
|------|--------|
| 任务完成度 | 是否真的完成了目标，而不是只给出看似合理的回答 |
| 稳定性 | 多次运行是否表现一致，是否容易中途跑偏 |
| 可控性 | 是否听得懂约束，出错后是否容易纠正 |
| 透明度 | 是否能展示关键过程、依据和改动 |
| 工程集成 | 是否适合接入代码仓库、文档、浏览器、命令行等真实环境 |
| 成本与速度 | 完成同类任务需要多少时间、调用和人工干预 |
| 安全边界 | 是否避免误删、泄露、越权和不可逆操作 |
| 可复现性 | 其他人是否能根据公开说明得到相近结果 |

## 如何阅读一份评测

建议按这个顺序看：

1. 先看任务说明：这个评测到底模拟了什么场景。
2. 再看成功标准：什么才算完成，什么只是“看起来完成”。
3. 然后看过程记录：工具是怎样计划、执行、修正的。
4. 最后看结论：适合谁，不适合谁，有哪些限制。

如果你只想快速了解一个工具，可以直接看工具档案和最终建议；如果你想认真选型，建议阅读完整任务和复现说明。

## 适合谁

- 想选择 AI Agent 工具，但不想只听宣传文案的人。
- 想把 Agent 用进真实工作流的开发者。
- 想理解 Codex、Claude Code、Cursor、Gemini CLI、OpenCode 等工具差异的人。
- 想复现评测、提出改进或贡献测试任务的人。

## 当前状态

项目刚开始搭建，首批内容会优先补齐：

- 基础评测方法
- 第一组公开评测任务
- 样例数据
- 工具档案模板
- 可复现报告格式

早期结论会比较谨慎，因为 Agent 工具变化很快。每份评测都应该带着日期、环境和限制条件来理解。

## 参与方式

你可以通过这些方式参与：

- 提出你希望评测的工具。
- 提交真实但可公开的任务场景。
- 指出评测设计里的漏洞。
- 复现已有评测并反馈差异。
- 改进文档、数据集或工具适配器。

如果你只是想跟着内容学习，也可以从 `docs/` 和后续发布的评测报告开始。

## 声明

本项目的评测结论只代表特定时间、任务和环境下的观察，不构成绝对排名。AI Agent 工具更新很快，建议结合你的真实场景做小规模验证后再决定是否长期使用。
